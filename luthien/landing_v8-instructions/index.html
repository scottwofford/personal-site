Luthien QA Trial

User onboarding test — walk through our product as a new user and report what's broken or confusing.

Goal: Follow the onboarding flow end-to-end. Don't fix anything — just document issues, confusion, and broken stuff. That's the deliverable.

Prerequisites

  • Mac or Linux machine
  • Docker Desktop installed and running
  • Anthropic API key (free tier works)
  • Node.js 18+ and Claude Code: npm install -g @anthropic-ai/claude-code

Part 1: Landing Page ~10 min

scottwofford.com/luthien/landing_v8/

Open the landing page and answer:

  1. First impression (30 sec): Without scrolling — what does this product do? Write your answer before scrolling.
  2. Carousel: Do the quotes load and animate? Any broken cards or layout issues?
  3. Click "more examples →": Does the incidents page load? Do filter pills work? Does infinite scroll work?
  4. Architecture diagram: Does it make sense? Could you explain Luthien to a coworker after reading it?
  5. CTAs: "Book a Call" opens Google Calendar? GitHub icon goes to the right repo?
  6. Mobile: Resize browser to phone width. Does everything still work?

Part 2: README Quick Start ~45–60 min

README.md (PR #179 branch)

2a. Read the intro sections ~5 min

Before installing anything, read from the top:

  1. "What is it?" — Clear?
  2. Before/After terminal mockups — Do the SVGs render?
  3. "How does it work?" — Does the ASCII diagram make sense?
  4. "What can it do?" — Do the expandable code examples expand?

2b. Follow the Quick Start exactly ~30–45 min

Follow every step exactly as written. Do NOT skip ahead or improvise. The point is to test whether a new user can follow these instructions cold.

For each step, note: Did it work as written? If not — exact error + screenshot. Anything confusing? How long did it take?

2c. Verify it works ~10 min

Once Claude Code is running through the proxy:

  1. Ask Claude Code to do something simple (e.g., "create a hello world python script")
  2. Check the conversation history UI — does your conversation show up?
  3. If you set up DeSlop: ask Claude to write something with em dashes or "I'd be happy to help" — does it get cleaned?
  4. Check the activity monitor — do you see requests flowing?

Part 3: Your Report

Create a Google Doc or markdown file with:

  1. Landing page issues (Part 1)
  2. README issues (Part 2a + 2b)
  3. Functionality issues (Part 2c)
  4. Overall impressions:
    • How long did the whole thing take?
    • On a scale of 1–5, how easy was it to get running?
    • What was the most confusing part?
    • What would you change first?

Send the report to scott@luthienresearch.org.

Tips:
  • If something is broken or confusing, that's valuable feedback — not a failure on your part.
  • Don't fix things yourself. Just document what's wrong.
  • If stuck on a step for >10 min, note where you got stuck, skip it, and keep going.
  • Docker builds images locally on first run — can be slow (~5–10 min). That's expected.
